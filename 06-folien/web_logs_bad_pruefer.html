M3.10 Systemintegration

Logs lesen wie ein Prüfer

Angriffs-Patterns erkennen, typische Fehler erklären und sinnvolle Maßnahmen ableiten.

Heute geht es nicht um tiefes Hacken, sondern um Verständnis: Was ist normal? Was ist verdächtig? Was ist kaputt?

Grundidee

Alles ist verdächtig, was nicht zum normalen Weg passt

Normal

Ein Formular sendet per POST an save.php. Danach kommt ein Redirect oder eine Erfolgsmeldung.

Verdächtig

Jemand ruft save.php per GET mit komischen Parametern auf oder testet viele seltsame URLs.

POST /save.php HTTP/1.1       → passt zum Formular
GET  /save.php?id=1 HTTP/1.1 → prüfen: Warum GET?
Grundlagen

Wie liest man eine Log-Zeile?

192.168.1.10 - - [29/Apr/2026:10:15:01] "GET /index.php HTTP/1.1" 200 "Mozilla/5.0"
  • IP: Wer greift zu?
  • Zeit: Wann passiert es?
  • Methode: GET oder POST?
  • URL: Welche Seite?
  • Status: Ergebnis (200, 403, 500 …)
  • Browser: echter Nutzer oder Bot?

Merksatz: Ein Log ist wie ein Kassenbon – da steht alles drauf.

Pattern 1

Falsche Methode: GET statt POST

Speichern, Ändern oder Löschen sollte nicht einfach per Link oder URL passieren.

GET /delete.php?id=42 HTTP/1.1 200
GET /save.php?name=Bob HTTP/1.1 200
  • Problem: Aktion kann durch Link, Bot oder Browser-Verlauf ausgelöst werden.
  • Maßnahme: Für Änderungen POST verwenden, Eingaben prüfen, ggf. Session/CSRF-Schutz.
Pattern 2

Komische Parameter in der URL

Angreifer testen oft, ob Parameter ungeprüft verwendet werden.

GET /list.php?id=1' HTTP/1.1 500
GET /list.php?id=1 OR 1=1 HTTP/1.1 500
GET /profile.php?user=<script>alert(1)</script> HTTP/1.1 200
  • Verdacht: SQL-Injection-Test oder XSS-Test.
  • Maßnahme: Prepared Statements, filter_input, Ausgaben mit htmlspecialchars entschärfen.
Pattern 3

Viele 404: jemand sucht Schwachstellen

Einzelne 404 sind normal. Viele typische Admin- oder Tool-Pfade sind auffällig.

GET /wp-admin/ HTTP/1.1 404
GET /phpmyadmin/ HTTP/1.1 404
GET /.env HTTP/1.1 404
GET /backup.zip HTTP/1.1 404
  • Bedeutung: Automatischer Scanner sucht bekannte Einstiegspunkte.
  • Maßnahme: Keine Backups öffentlich ablegen, unnötige Tools entfernen, Server aktuell halten.
Pattern 4

403: Zugriff wurde blockiert

403 ist nicht automatisch schlecht. Oft ist es ein gutes Zeichen: Der Server verweigert Zugriff.

GET /admin/ HTTP/1.1 403
GET /config.php HTTP/1.1 403
GET /.git/config HTTP/1.1 403
  • Interpretation: Ressource existiert eventuell, aber Zugriff ist verboten.
  • Prüffrage: Sollte diese Ressource überhaupt im Web-Verzeichnis liegen?
Pattern 5

500: interner Fehler – für Angreifer interessant

500 bedeutet: Auf dem Server ist etwas kaputt gegangen. Das kann Hinweise auf unsauberen Code geben.

GET /list.php?id=abc HTTP/1.1 500
[error] PHP Fatal error: Uncaught PDOException...
  • Problem: Fehler verraten eventuell Technik, Dateipfade oder SQL-Probleme.
  • Maßnahme: Saubere Fehlerbehandlung, keine Debug-Ausgaben für Benutzer, error.log prüfen.
Pattern 6

Überlastungsversuch: sehr viele Anfragen

Umgangssprachlich wird so etwas manchmal „Ping of Death“ genannt. In Web-Logs sieht man eher das Muster: extrem viele Requests in kurzer Zeit.

GET /index.php HTTP/1.1 200
GET /index.php HTTP/1.1 200
GET /index.php HTTP/1.1 200
GET /search.php?q=test HTTP/1.1 200
GET /search.php?q=test HTTP/1.1 200
... viele hundert Einträge pro Minute von derselben IP ...
  • Verdacht: Der Server soll überlastet werden, z. B. durch einen DoS-Angriff.
  • Erkennbar an: Viele gleiche oder ähnliche Anfragen in sehr kurzer Zeit.
  • Maßnahme: Rate Limiting, Firewall, Monitoring, Logs auswerten, auffällige IPs blockieren.
Merksatz

access.log zeigt: Was wurde angefragt?

error.log zeigt: Was ist kaputt gegangen?

access.log

Methode, URL, Statuscode, Zeitpunkt, Browser/IP.

error.log

PHP-Fehler, Serverprobleme, Zertifikatswarnungen, interne Details.

Bad Prüfer 1 – Aufgabe

Was ist hier verdächtig?

GET /save.php?name=Tom&email=test@example.com HTTP/1.1 200

Ein Schüler sagt: „Status 200 heißt, alles ist in Ordnung.“ Stimmt das?

Bad Prüfer 1 – Lösung

200 heißt nur: Server hat geantwortet

  • Verdächtig: save.php wird per GET aufgerufen.
  • Warum? Speichern sollte normalerweise per POST erfolgen.
  • Maßnahme: Formularmethode prüfen, serverseitig Methode validieren.
if ($_SERVER['REQUEST_METHOD'] !== 'POST') { http_response_code(405); }
Bad Prüfer 2 – Aufgabe

Was könnte der Angriff sein?

GET /list.php?id=1' HTTP/1.1 500
GET /list.php?id=1 OR 1=1 HTTP/1.1 500

Erkläre kurz, was getestet wird und welche Schutzmaßnahme passt.

Bad Prüfer 2 – Lösung

Verdacht: SQL-Injection-Test

  • Das Hochkomma testet, ob SQL-Abfragen kaputt gehen.
  • OR 1=1 testet, ob Bedingungen manipuliert werden können.
  • Schutz: PDO Prepared Statements, Datentypen prüfen, keine SQL-Strings zusammenkleben.
Bad Prüfer 3 – Aufgabe

404: Fehler oder Angriff?

GET /wp-admin/ HTTP/1.1 404
GET /.env HTTP/1.1 404
GET /backup.zip HTTP/1.1 404

Warum ist das trotz 404 wichtig?

Bad Prüfer 3 – Lösung

Viele typische Pfade = Scanner

  • 404 bedeutet: Die Datei wurde nicht gefunden.
  • Aber: Die Pfade sind typische Ziele automatischer Scans.
  • Maßnahme: Keine geheimen Dateien im Webroot, Updates, Rechte prüfen, Logs beobachten.
Bad Prüfer 4 – Aufgabe

XSS oder nicht?

GET /profile.php?name=<script>alert(1)</script> HTTP/1.1 200

Welche zwei Stellen muss man unterscheiden?

Bad Prüfer 4 – Lösung

Eingabe prüfen, Ausgabe entschärfen

  • Eingabe: filter_input hilft beim Einlesen und Validieren.
  • Ausgabe: htmlspecialchars verhindert, dass HTML/JS als Code ausgeführt wird.
  • Wichtig: Wenn das Script schon vorher im HTML stand, hilft ein späterer echo nicht mehr.
Bad Prüfer 5 – Aufgabe

Was verrät das error.log?

[error] PHP Warning: Undefined array key "email" in /var/www/html/save.php on line 12
[error] PHP Fatal error: Uncaught PDOException: SQLSTATE...

Welche typischen Entwicklerfehler erkennt man?

Bad Prüfer 5 – Lösung

Ungeprüfte Eingaben + DB-Fehler

  • Undefined array key: Formularfeld wurde erwartet, aber nicht sicher geprüft.
  • PDOException: Datenbankabfrage oder Verbindung ist fehlgeschlagen.
  • Maßnahme: isset/filter_input nutzen, Fehler behandeln, Benutzer nicht mit Debug-Details konfrontieren.
Bad Prüfer 6 – Aufgabe

Überlastung oder normaler Besuch?

10:15:01 GET /index.php HTTP/1.1 200
10:15:01 GET /index.php HTTP/1.1 200
10:15:02 GET /index.php HTTP/1.1 200
10:15:02 GET /index.php HTTP/1.1 200
10:15:03 GET /index.php HTTP/1.1 200
... sehr viele weitere Einträge von derselben IP ...

Warum kann das trotz Status 200 ein Sicherheitsproblem sein?

Bad Prüfer 6 – Lösung

Viele 200er können trotzdem gefährlich sein

  • 200 bedeutet: Die Seite wurde erfolgreich ausgeliefert.
  • Aber: Sehr viele Anfragen können CPU, RAM, Datenbank oder Netzwerk belasten.
  • Verdacht: DoS- oder Überlastungsversuch.
  • Maßnahme: Rate Limiting, Monitoring, Firewall-Regeln, auffällige IPs prüfen.
Mini-Checkliste

So analysieren Teilnehmer Logs

1

Methode

GET oder POST? Passt das zur Aktion?

2

URL

Normale Seite oder komische Parameter/Pfade?

3

Status

200, 400, 403, 404, 500 richtig einordnen.

4

Fehler

Steht im error.log etwas zu PHP, SQL oder Server?

5

Risiko

Kann jemand Daten lesen, ändern oder Code ausführen?

6

Maßnahme

Validierung, Prepared Statements, Sessions, Rechte, Updates.

Abschluss

Prüfungsnah denken

Nicht auswendig lernen: lesen, einordnen, begründen.

  • Was ist der normale Ablauf?
  • Welche Logzeile passt nicht dazu?
  • Welche Sicherheitsmaßnahme verhindert den Fehler?
