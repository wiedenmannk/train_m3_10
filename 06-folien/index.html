Nicht primär programmieren – sondern verstehen, prüfen, einordnen und Fehler erkennen.

Praxis als Werkzeug

Praxis hilft beim Verstehen: Durch eigene Beispiele und kleine Programme wird klar, wie Webanwendungen intern arbeiten.

Wir nutzen Programmierung als Werkzeug, um Abläufe zu verstehen und Systeme besser einordnen zu können.

Der Grundablauf einer Webanwendung

Dieser Ablauf ist der rote Faden für fast alle Themen in M3.10.

Benutzer→Browser→Request→PHP / Server→Datenbank→Response 
Benutzer klickt auf Senden
→ Browser sendet Formular-Daten
→ PHP verarbeitet
→ Datenbank wird gelesen oder geschrieben
→ Webseite antwortet

Formulare lesen und prüfen

Formularattribute bestimmen, wohin Daten gesendet werden und wie die Übertragung abläuft.

<form action="save.php" method="post" autocomplete="on">
  <label for="name">Name</label>
  <input id="name" name="name">
  <button>Senden</button>
</form>
AttributBedeutung
actionZieladresse, z. B. save.php
methodÜbertragungsart, meist get oder post
autocompleteBrowser darf Eingaben vorschlagen
novalidateBrowser-validierung wird abgeschaltet – Server muss trotzdem prüfen

enctype: wichtig bei Datei-Uploads

Das Attribut bestimmt, wie Formulardaten kodiert übertragen werden.

Normales Formular

<form method="post">

Für normale Textfelder reicht das meistens.

Datei-Upload

<form method="post"
      enctype="multipart/form-data">
  <input type="file" name="upload">
</form>

Ohne diesen Wert kommt die Datei typischerweise nicht korrekt beim Server an.

SI-Prüffrage: „Warum funktioniert der Upload nicht?“ – möglicher Grund: falsches oder fehlendes enctype.

GET vs. POST

Einer der wichtigsten Prüfungsbereiche.

FrageGETPOST
Wo stehen Daten?In der URLIm Request-Body
Sichtbar?Ja, z. B. ?name=BobNicht direkt in der URL
Cache/Bookmark?Eher jaEher nein
Typischer ZweckLesen, Suchen, FilternSpeichern, Ändern, Login, Löschen
Seiteneffekte?Darf keine Daten verändernDarf Daten verändern, z. B. speichern oder löschen

Merksatz: GET = holen/anzeigen. POST = senden/verändern.

Hinweis: In modernen Web-APIs wird für das Löschen oft die HTTP-Methode DELETE verwendet. In klassischen Formularen wird dafür meist POST genutzt.

SI-Fokus: Methode, Ziel, Statuscode und Datenfluss erkennen.

PHP/PDO lesen: Was passiert hier?

Formulardaten werden aus dem Request gelesen, geprüft oder bereinigt und anschließend sicher an die Datenbank übergeben.

$name = filter_input(INPUT_POST, 'name', FILTER_SANITIZE_SPECIAL_CHARS);

$stmt = $pdo->prepare('INSERT INTO kontakte (name) VALUES (:name)');
$stmt->execute(['name' => $name]);

filter_input

Liest Eingaben aus Request-Daten, z. B. aus POST, und kann Werte validieren oder bereinigen.

prepare + execute

prepare() bereitet den SQL-Befehl mit Platzhaltern vor. execute() übergibt die echten Werte sicher.

SQL Injection erkennen

Schutz vor SQL Injection: Benutzereingaben werden sicher über Parameter übergeben (Prepared Statements) und nicht direkt in SQL-Befehle eingebaut.

Unsicher: Eingaben werden direkt in SQL eingefügt.
Sicher: Werte werden getrennt übergeben und nicht als SQL-Code interpretiert.

Unsicheres Muster

$sql = "SELECT * FROM users
        WHERE name = '$name'";

Eingabe wird direkt in SQL eingebaut.

Sicheres Muster

$stmt = $pdo->prepare(
  'SELECT * FROM users WHERE name = :name'
);
$stmt->execute(['name' => $name]);

Wert wird als Parameter übergeben.

Prüfungsantwort: Prepared Statements schützen vor SQL Injection, weil Eingaben nicht als SQL-Code interpretiert werden.

Validierung vs. Sanitizing

Beide Begriffe werden gerne verwechselt.

Validierung

Prüfen: Ist die Eingabe gültig?

Ist das eine E-Mail?
Ist die Zahl im erlaubten Bereich?

Sanitizing

Bereinigen: Gefährliche oder unerwünschte Zeichen entschärfen.

HTML-Zeichen entfernen oder entschärfen

Wichtig: Client-Prüfung hilft dem Benutzer. Server-Prüfung ist Pflicht.

Sichere Ausgabe: XSS vermeiden

Daten aus Formularen oder Datenbanken dürfen nicht ungeprüft als HTML ausgegeben werden.

Problem

<script>alert('XSS')</script>

Kann als Script ausgeführt werden, wenn es ungefiltert in HTML landet.

Entschärfte Ausgabe

echo htmlspecialchars($name);

HTML-Sonderzeichen werden als Text angezeigt.

SI-Fokus: Eingabeprüfung und sichere Ausgabe gehören zusammen.

Passwort-Sicherheit

Passwörter werden nicht im Klartext gespeichert.

Falsch

passwort = "geheim123"

Klartext in der Datenbank ist ein schweres Sicherheitsproblem.

Richtiges Prinzip

password_hash()
password_verify()

Beim Login wird nicht entschlüsselt, sondern geprüft, ob das eingegebene Passwort zum Hash passt.

Übertragung beachten

Ohne HTTPS werden Daten (auch Passwörter) unverschlüsselt übertragen und können im Netzwerk mitgelesen werden.

Deshalb müssen Login-Formulare immer über HTTPS gesendet werden.

Mindestanforderung: Hashing mit geeigneten Verfahren wie bcrypt oder Argon2; keine Klartextspeicherung.

Sessions: Login-Zustand merken

HTTP ist grundsätzlich zustandslos. Sessions helfen, einen Benutzer über mehrere Requests wiederzuerkennen.

Eine Session speichert Daten auf dem Server, um einen Benutzer über mehrere Anfragen hinweg wiederzuerkennen. Beispiel: Nach dem Login merkt sich der Server, dass der Benutzer eingeloggt ist.

Ohne Session müsste der Zustand immer wieder neu übertragen werden, z. B. über Formularfelder oder URL-Parameter. Das wäre unpraktisch und kann Sicherheitsprobleme verursachen.

Login→Session-ID→Server merkt Zustand→geschützter Bereich

SI-Mindestanforderungen

Session-Cookies schützen, Login-Zustand sauber beenden, sensible Bereiche nicht ohne Prüfung anzeigen.

Maßnahme

AP1 vs. AP2 für SI

AP1: Grundlagenebene

Web-Prinzipien, HTML/CSS-Semantik, GET vs. POST, einfache Statuscodes und Log-Hinweise verstehen.

AP2: Lesefähigkeit & Prüfung

Logs interpretieren, Formular-/Header-Aspekte nachvollziehen, Sicherheitsmuster wie Prepared Statements, Hashing und Sessions erkennen.

Wichtige Fähigkeiten: Webabläufe verstehen, GET/POST unterscheiden, Logs einordnen und Sicherheitsaspekte erkennen.

Prüfungs-Merksatz

SI liest den Weg der Daten: Formular → Request → Server → Datenbank → Response → Log.

Und erkennt dabei Fehler, Risiken und sinnvolle Maßnahmen.
